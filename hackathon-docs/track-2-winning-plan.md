# Track 2 Winning Plan: Cross-Model Spec Comparison

## Goal

Build a small, credible research artifact for Track 2, Specification Validation, that compares Lean 4 specification candidates generated by two OpenAI models, surfaces where they disagree, and demonstrates concrete counterexamples for the highest-value disagreements.

The entry must be fast to execute, easy to explain, and strong enough to read like a real research prototype rather than a demo.

## Core Idea

Generate three spec candidates per task, using two different OpenAI models, then normalize every output into a Lean 4-centered canonical form. Compare the normalized specs structurally and semantically, identify divergences, and use executable validators plus Hypothesis fuzzing to produce counterexamples that show which interpretation is too weak, too strong, or otherwise inconsistent.

This is a specification-validation project, not a code-generation project. The novelty is in treating disagreement between model-produced specs as the signal, then turning that disagreement into concrete evidence.

## Working Assumptions

- Lean 4 will be used as the target syntax for canonical spec fragments, but the validation and fuzzing loop can fall back to Python when Lean execution is too slow or too brittle for the time available.
- The report will prioritize depth over breadth. Three tasks are enough if each yields clear disagreements and at least one concrete counterexample.
- Because of the budget, prompts must be short, outputs tightly bounded, and each call must be structured to minimize token waste.

## Task Set

Use three tightly scoped tasks that are easy to specify and easy to falsify:

1. Sortedness property.
   - Input: list of comparable values.
   - Candidate specification question: what properties must a sorting function satisfy beyond simply returning a list?
   - Likely disagreement zones: stability, permutation preservation, duplicate handling, emptiness.

2. Token-bucket rate limiter semantics.
   - Input: request stream with timestamps.
   - Candidate specification question: what counts as refill, overflow, and rejection behavior?
   - Likely disagreement zones: refill ordering, initial capacity, rounding, burst handling.

3. Token expiry / validation semantics.
   - Input: token issue time, expiry time, now, and optional clock skew.
   - Candidate specification question: when should a token be accepted or rejected?
   - Likely disagreement zones: inclusive vs exclusive expiry, skew tolerance, revocation assumptions.

## Method

### 1. Prompting

Use one prompt family for each model, but keep the task statement identical so the disagreement comes from model behavior rather than task drift.

Each generation request should ask for:
- A Lean 4 specification fragment.
- A short structured explanation of assumptions.
- A minimal example or edge case.
- No prose beyond what is necessary to support the spec.

The prompts should force the model to commit to explicit semantic choices that can later be compared.

### 2. Canonicalization

Parse each response into a canonical Lean-oriented record with fields like:
- `assumptions`
- `preconditions`
- `postconditions`
- `invariants`
- `examples`
- `notes`

The canonical form does not need to prove equivalence. It only needs to make differences easy to inspect and compare.

### 3. Disagreement Detection

Detect disagreement in three layers:
- Structural: fields are present, omitted, or contradictory.
- Semantic: one spec allows behaviors the other excludes.
- Empirical: executable validators and fuzzing find an input accepted by one spec and rejected by the other.

### 4. Counterexample Generation

For each disagreement, try to produce a concrete witness using Hypothesis.

The ideal counterexample is small, readable, and directly tied to the competing semantic claims.

### 5. Reporting

Produce a static HTML page that shows:
- task name
- all six spec samples per task
- the canonical fields after normalization
- highlighted disagreements
- a counterexample section with executable witness code

Then write a concise PDF report using the hackathon template.

## Budget Plan

The budget is the tightest constraint, so use a deliberately small call budget.

Call count target:
- 3 tasks
- 3 samples per task
- 1 sample per task from `gpt-4o`
- 2 samples per task from `gpt-3.5-turbo`
- Total: 9 spec-generation calls

Recommended limits:
- Keep each prompt short.
- Keep each response short.
- Use low temperatures, ideally between 0.0 and 0.2.
- Cap response length aggressively so the model outputs only what is needed for comparison.

## Success Criteria

The entry is good enough if it achieves all of the following:

- It compares two OpenAI models on the same spec task and makes the disagreement visible.
- It uses Lean 4 as the canonical spec anchor.
- It demonstrates at least one concrete counterexample per task or per major disagreement cluster.
- It is small enough to finish tonight and polished enough to submit as a research artifact.
- It clearly states limitations, especially where the method cannot establish true semantic equivalence.

## Risks

1. Budget overrun.
   - Mitigation: keep the task set at three, keep outputs short, and minimize retries.

2. Spec outputs are too verbose or too unstructured.
   - Mitigation: tighten prompts and require a fixed output schema.

3. Lean parsing is brittle.
   - Mitigation: treat Lean as the canonical representation, but allow Python validators as the executable backstop.

4. No strong disagreements appear.
   - Mitigation: choose tasks with ambiguous boundary semantics and compare multiple samples per model.

## Research Angle

The report should emphasize that the project is not merely comparing model answers. It is testing whether model-generated specifications actually converge on a coherent behavior contract. The key claim is that cross-model disagreement is a practical proxy for underspecification, ambiguity, or hidden assumptions.

That gives the project a clean theory of change:
- generate two spec candidates
- normalize them
- surface disagreements
- probe them with executable tests
- turn ambiguity into a concrete witness

## Final Submission Shape

The submission should look like a compact research paper with these sections:
- introduction
- related work
- methods
- results
- discussion and limitations
- conclusion
- references
- appendix

The static HTML artifact is for inspection and demo value, but the PDF is the actual submission. The HTML should support the PDF by making the disagreements easy to inspect.
